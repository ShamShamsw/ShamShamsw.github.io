Serverless Order Processing System Project

Efficient, scalable, and cloud-powered order management.

.

Description

The Serverless Order Processing System is a cloud-based solution designed for efficient and scalable order management. When a user places an order on the website, the frontend sends an HTTP request to Amazon API Gateway, which acts as the secure entry point to the backend system. The API Gateway forwards the order details to the Order Processing Lambda, which contains the core business logic and orchestrates the steps required to process the order, such as validating stock, handling payment, and confirming the order via an SNS notification or email. The Order Processing Lambda triggers an AWS Step Functions workflow, breaking the process into smaller, modular steps for better management and scalability. The first step ensures product availability by calling a Validate Stock Lambda, which queries Amazon DynamoDB to check inventory levels. If sufficient stock is available, the workflow proceeds; otherwise, it halts or branches based on the error. Additional AWS services include:

  • AWS Cognito: Manages user authentication, handling sign-ups, logins, and identity management.
  • AWS IAM: Controls access permissions, ensuring secure communication between services.
The backend logic is implemented through specialized Lambda functions:
  • Get Order: Retrieves order details from the database.
  • Allocation for Rules: Applies business rules to determine how orders should be allocated (e.g., prioritization, splitting).
  • Carrier Data: Retrieves shipping options, rates, and availability from integrated carriers.
  • Facility Data: Provides real-time data about warehouse or store availability, including stock levels and processing capacity.
The final part of the Application Tier integrates tightly with the Data Layer, distributed across Availability Zones 1 and 2 to ensure high availability, fault tolerance, and performance. The system uses Amazon DynamoDB as the primary data store for both order and product information, with a write and read replica setup for optimal efficiency:
  • In Availability Zone 1, each DynamoDB table includes a write replica, handling all data creation and modification tasks such as placing new orders, updating stock levels, or modifying product details.
  • In Availability Zone 2, each table has a read replica, optimized for high-speed query access, such as retrieving order history, inventory levels, or product catalog details.
This architecture ensures that the system remains highly available and fault-tolerant, even in the event of an outage in one availability zone. By leveraging DynamoDB's global table capabilities, the system can maintain consistent performance and data integrity across distributed environments, providing a seamless experience for users.
